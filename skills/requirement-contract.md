# 需求契约

## 职责

在需求进入路线决策、原型评审、任务拆解或 OpenSpec 之前，确保它有一份写下来的、已确认的"需求契约"。

区分两件事：

- **必经的**：需求契约本身（brief 或 PRD 二选一）。
- **不必经的**：完整 PRD。大多数个人项目、小修小补、低风险需求，一份 5-10 行的 brief 就够，不需要走完整 PRD 的分段生成流程。

## 两种模式

| 模式 | 适用场景 | 产物形态 |
|------|---------|---------|
| `brief`（默认） | 个人项目、小修小补、老项目叠加功能、低风险低复杂度需求 | 5-10 行：目标、范围、不做什么、验收标准、风险/待确认 |
| `prd`（复杂模式） | 产品型界面、多状态 UI、多角色协作、涉及权限/计费/敏感信息、团队协作/团队多环境项目、用户明确要求 PRD | 分模块生成，逐段人工确认：背景与目标 → 用户/场景 → 功能范围 P0/P1/P2 → 业务规则与边界 → 验收标准 |

模式选择由调用方（通常是 `sdd-router`）根据已有的项目协作模式、需求形态、复杂度/风险判断结果建议，用户可以覆盖。拿不准时，先建议 `brief`，需要时可以从 brief 升级为 PRD，不强制一步到位选重的。

## 输入优先级

1. **会议纪要 / 需求沟通对话记录**（优先）——信息纯净、只记双向确认后的结论
2. **零散文字描述**——用户随手写的几句话、聊天记录片段
3. **用户口头补充**——AI 主动追问缺口

**不要求用户先写一份 BRD 再喂给你**。BRD 式输入噪音多（背景目标混杂、方案越界、共识模糊），是这个 Skill 明确不采用的输入方式。

如果用户已经有 PRD、brief 或外部需求文档链接，直接引用它、不重新起草；只需要向用户确认"这份还是当前有效的吗"，确认后即可视为已确认契约，跳过生成步骤。

## brief 模式

### 产物结构

```markdown
# <需求名> · 需求 Brief

> 状态：草稿 / 待确认 / 已确认
> 日期：YYYY-MM-DD
> 输入来源：会议纪要 / 对话描述 / 口头补充

- **目标**：{一句话，这个需求要解决什么}
- **范围**：{本轮做什么，边界在哪}
- **不做什么**：{明确排除的内容，避免范围蔓延}
- **验收标准**：{优先写成 Given / When / Then，见下方规则；简单场景可写一句"怎么算做完"}
- **假设与默认值**：{信息不足、追问已达上限时，AI 假设的默认取值，需用户事后 review；无则写"无"}
- **风险/待确认**：{不确定的点，需要谁来拍板}
```

控制在 5-10 行（不含 GWT 展开），不展开成完整 PRD 的章节结构。如果发现内容自然膨胀超出这个量级，说明这个需求可能需要升级到 `prd` 模式，向用户提出确认。

### 输出位置

```text
<docs_dir>/Requirements/YYYY-MM-DD · <需求名> brief.md
```

## 验收标准：优先 Given / When / Then

验收标准优先用 BDD 三段式写，让后续 spec-compliance-check 和 TDD 能直接派生测试矩阵，不用把自由文字二次翻译成测试。

格式：

```markdown
- **验收标准**：
  - Given {前置状态/上下文}，When {触发动作}，Then {可观察的预期结果}
  - Given {前置}，When {动作}，Then {预期}
  - （异常/边界）Given {异常前置}，When {动作}，Then {错误处理预期}
```

规则：

- 每条 AC 是一个可独立验证的行为，Then 必须是可观察、可断言的结果，不写"体验更好"这类不可测表述。
- 尽量覆盖正常路径 + 至少一个异常/边界场景。
- 纯配置、纯文案、纯样式等确实无行为可断言的小改动，可以退回一句话验收标准，但要说明"无可派生测试场景"。
- brief 和 prd 模式的验收标准都适用本规则。

## 反问上限与假设默认值

需求信息不足时可以追问，但不允许无限追问，避免"十几轮反问、每轮都差不多"的死循环。

规则：

- **最多反问 2 轮**。每轮把关键缺口一次问清，不要挤牙膏式一次只问一个。
- 到第 3 轮仍有缺口时，**不再追问**，改为对剩余缺口取合理默认值，写进契约的"假设与默认值"段落，交由用户事后 review 覆盖。
- 用户给出 escape 信号（如"够了"、"按你想的来"、"你看着办"、"先这样"）时，**立即停止追问**，把未确认项写入"假设与默认值"，进入待确认状态。
- 假设默认值必须显式标注为 AI 假设，不能伪装成用户已确认的结论。
- 反问只针对影响范围、验收标准、边界的关键缺口；无关紧要的细节直接取默认值，不占用反问轮次。

## prd 模式

### 分段生成规则

**不要一次性生成整份 PRD。** 按以下顺序逐段生成，每段生成后停下来，等用户确认这一段，确认后才生成下一段：

1. 背景与目标
2. 用户 / 场景
3. 功能范围（P0 / P1 / P2 分级）
4. 业务规则与边界
5. 验收标准

每段生成前，先说明这段准备写什么、依据是什么（来自会议纪要哪部分、还是 AI 推断），推断内容必须显式标注。

### 上下文投喂原则

不要把项目所有历史文档全量塞进上下文。优先给精炼摘要：本轮相关的 `Decisions/` 结论、同类历史需求的关键结论、相关代码模块索引（如果有）。全量堆料容易导致内容编造或跑题，只投喂和本次需求直接相关的精炼材料。

### 产物结构

```markdown
# <需求名> · PRD

> 状态：草稿 / 待确认 / 已确认
> 日期：YYYY-MM-DD
> 输入来源：会议纪要 / 对话描述 / 口头补充
> 生成方式：分段生成，逐段人工确认

## 一、背景与目标

## 二、用户 / 场景

## 三、功能范围

### P0（必须）

### P1（重要）

### P2（可选）

## 四、业务规则与边界

## 五、验收标准
（优先 Given / When / Then 三段式，见"验收标准：优先 Given / When / Then"节）

## 六、假设与默认值
（反问达上限或用户 escape 后，AI 取的默认取值，需用户事后 review；无则写"无"）

## 七、待确认问题
```

### 输出位置

```text
<docs_dir>/PRDs/YYYY-MM-DD · <需求名> PRD.md
```

## 状态标记

文件顶部必须标注三种状态之一：

- **草稿**：AI 生成中，尚未经过用户确认
- **待确认**：AI 已生成完整内容（brief 全文，或 PRD 全部段落），等待用户逐条/逐段过一遍
- **已确认**：用户明确确认内容准确，可以作为后续阶段的输入

## Gate 规则

**只有状态为"已确认"的需求契约，才能进入 `route-decision-recorder`、手动原型评审记录、`task-planning` 或 OpenSpec。**

状态还是"草稿"或"待确认"时，停在这一步，不要替用户默认确认，也不要自行推进到下一阶段。

## 移交规则

契约确认后，根据 `sdd-router` 传入的模式判断和扩展开关移交：

- **完整模式 + 需要产品原型评审** → 移交 `route-decision-recorder`，路线决策文档中引用本契约路径；v0.1 先记录评审范围和待确认问题
- **完整模式 + 不需要产品原型评审** → 移交 `route-decision-recorder`，引用本契约路径
- **轻量模式** → 移交 `task-planning`，引用本契约路径（`brief` 通常已经足够）
- **用户明确要求先看原型** → 可在契约确认后先记录手动原型评审范围，`prd` 模式产出的 PRD 可作为 PRD-driven 输入

移交时传递：契约文件路径、模式（brief/prd）、已确认结论摘要、待确认问题（如有遗留）。

**原型评审不承担 PRD 起草逻辑**——它只消费已确认的契约（brief 或 PRD）生成可视化原型评审上下文，起草工作在这个 Skill 里完成，不重复实现。自动 `proto-review` 后续提供，不在 v0.1 发布范围内。

## 纠错处理

需求契约起草过程中如果用户纠正了 AI 的理解（比如"这个范围理解错了"、"这条业务规则不对"），**不建立独立的错题本**，按现有 `learn` Skill 的候选沉淀流程处理：整理成候选 lesson，等 session 收尾或用户主动确认后写入。

## 原则

- **必经的是契约，不是 PRD**：多数需求走 brief 就够，不要把 PRD 当成默认动作。
- **不要求用户先写 BRD**：优先用会议纪要或对话记录作为输入，用户口头补充作为兜底。
- **验收标准优先 GWT**：能写成 Given/When/Then 就写成三段式，方便下游直接派生测试；无行为可断言的小改动才退回一句话。
- **反问最多 2 轮**：信息不足时最多问 2 轮，之后取默认值写进"假设与默认值"，用户 escape 立即收尾，绝不无限追问。
- **假设显式标注**：AI 取的默认值必须标为假设，不能伪装成用户已确认结论。
- **分段生成，逐段确认**：PRD 模式绝不一次性生成整份文档。
- **精炼投喂，不堆料**：只给和本次需求相关的精炼上下文，不全量塞历史文档。
- **已有输入不重新起草**：用户已有 PRD/brief/外部链接时，引用并确认即可，不替用户重写。
- **状态是硬 Gate**：草稿/待确认状态不能移交下一阶段，这是本 Skill 唯一的强制点（GWT、反问上限、假设默认值都是软规则，不改变这个唯一硬 gate）。
- **纠错复用 learn**：不新建平行的经验沉淀机制。

## 常见误区

| 误区 | 问题 | 正确做法 |
|------|------|---------|
| 所有需求都走 PRD 模式 | 给小改动增加不必要的负担，违背"必经契约不等于必经 PRD"的设计初衷 | 默认 brief，只有命中 PRD 信号才升级 |
| 要求用户先写 BRD 再喂给 AI | 复刻了已验证效果更差的输入路径 | 优先用会议纪要/对话记录，缺失时才让用户口头补充 |
| PRD 一次性整份生成 | 幻觉严重、内容跑题 | 分段生成，每段停下来等确认 |
| 契约还在"待确认"就移交下一阶段 | 破坏了这个 Skill 唯一的强制点 | 严格执行 Gate，未确认不移交 |
| 给纠错单独建文件 | 和 `learn` 的经验沉淀机制重复 | 纠错走 `learn` 候选流程 |
| 反复追问需求细节十几轮 | 拖慢主路径的死循环 | 最多 2 轮，之后取默认值写进"假设与默认值"，用户 escape 立即收尾 |
| 验收标准写成"体验更好/更流畅" | 不可断言，下游无法派生测试 | 用 Given/When/Then 写可观察的预期结果 |
